## Сводка критических замечаний из аудитов

Оба аудита сходятся в следующих фундаментальных проблемах:

1. **Глобальный толеранс 1e-6** — не масштабируется, не учитывает размер модели, нет иерархии
2. **Триангуляция не гарантирует водонепроницаемость** — общие рёбра граней дискретизируются независимо, слияние постфактум
3. **Нет модуля healing** — «лечение» сводится к удалению, а не восстановлению
4. **Слабая поддержка вырожденных случаев** — NaN/Inf, нулевые рёбра, полюса, апексы
5. **STEP-парсер не валидирует геометрию** — нет семантической проверки, нет извлечения толерансов из файла
6. **Нет адаптивной триангуляции** — `max_deviation` и `max_edge_length` объявлены, но не используются
7. **Нет аналитических производных NURBS** — нормали вычисляются численно, нет dS/du, dS/dv
8. **Нет аналитических PCURVE** — только полилинейная аппроксимация, нет Curve2d
9. **Отсутствие многоуровневой валидации топологии** — нет проверки манифольдности, ориентации, замкнутости
10. **Нет распараллеливания** — полностью однопоточная обработка

## Фаза 1: Фундамент — Толерансы и Водонепроницаемость (0–3 месяца)

> **Цель:** Устранить корневые причины негерметичных сеток и некорректной дискретизации.

### 1.1 Система контекстных толерансов

**Проблема:** Глобальные константы `TOLERANCE=1e-6`, `ANGULAR_TOLERANCE=1e-5`, `PARAMETRIC_TOLERANCE=1e-8` не адаптируются к масштабу модели.

**Задачи:**

      pub relative: f64,       // Относительная погрешность (1e-8)
      pub angular: f64,        // Угловая погрешность в радианах (1e-5)
      pub parametric: f64,     // Параметрическая погрешность (1e-8)
      pub model_scale: f64,    // Характерный размер модели (bounding box diagonal)
  }
  ```
- [ ] 1.1.2 Метод `is_coincident(a, b) -> bool` с учётом `absolute + relative * model_scale`
- [ ] 1.1.3 Метод `parametric_tolerance(surface, u, v) -> f64` — пересчёт 3D-толеранса в UV-пространство с учётом якобиана поверхности (первая фундаментальная форма)
- [ ] 1.1.4 Метод `from_step_model(solid)` — автоматическое вычисление `model_scale` по диагонали bounding box

- [ ] 1.1.6 Заменить все использования `TOLERANCE` на `tol_ctx.absolute()` (или `tol_ctx.is_coincident()`)
- [ ] 1.1.7 Передавать `ToleranceContext` через всю цепочку: `StepConverter → triangulate_solid → triangulate_face → merge_coincident_vertices`

### 1.2 Кэш рёберной дискретизации (Edge Discretization Bus)

**Проблема:** Каждая грань триангулируется независимо, общие рёбра дискретизируются дважды с разными точками, потом сливаются `merge_coincident_vertices` — но это не гарантирует водонепроницаемость.

  }
  pub struct EdgeDiscretization {
      pub points_3d: Vec<Point3d>,    // 3D-точки на кривой
      pub points_uv: Vec<HashMap<TopoId, Point2d>>, // UV для каждой инцидентной грани
      pub params: Vec<f64>,            // Параметры кривой
  }
  ```
- [ ] 1.2.2 Метод `discretize_edge(edge, face_id, surface, tol_ctx)`:
  - Если ребро уже в кэше → вернуть существующие точки + вычислить UV для данной поверхности

- [ ] 1.2.3 Адаптивная дискретизация рёбер:

## Фаза 2: Триангуляция — Адаптивность и Корректность (3–6 месяцев)

> **Цель:** Перейти от фиксированной сетки к адаптивной триангуляции с контролем отклонения.

### 2.1 Адаптивная триангуляция на основе кривизны

**Проблема:** `TriangulationParams` содержит `max_deviation` и `max_edge_length`, но они не используются. Все криволинейные поверхности используют фиксированную сетку `angular_samples × height_samples`.

**Задачи:**

- [ ] 2.1.1 Реализовать вычисление кривизны для каждого типа поверхности:
  - `Plane`: κ = 0 (всюду)
  - `Cylinder`: κ₁ = 0 (образующая), κ₂ = 1/R (окружность)
  - `Cone`: κ₁ = 0, κ₂ = cos(α)/r(u)
  - `Sphere`: κ₁ = κ₂ = 1/R
  - `Torus`: κ₁ = cos(v)/(R + r·cos(v)), κ₂ = 1/r
  - `Nurbs`: численно через 1-ю и 2-ю фундаментальные формы
- [ ] 2.1.2 Метод `required_samples(surface, u_range, v_range, max_deviation) -> (usize, usize)`:
  - Вычислить максимальную кривизну в области
  - Количество сэмплов = ceil(arc_length / chord_step), где chord_step вычислен из max_deviation и κ
  - Формула: `chord_step = 2 * sqrt(2 * R * max_deviation)` для окружности радиуса R
- [ ] 2.1.3 Заменить фиксированные `angular_samples`/`height_samples` на вычисленные из `max_deviation`
- [ ] 2.1.4 Для плоскостей: всегда минимальное число треугольников (fan/ear-clipping)
- [ ] 2.1.5 Добавить контроль углового отклонения (angular deviation между нормалями соседних треугольников):
  - Если угол > `angular_tolerance` → subdividing
  - Это устраняет «фасеточный» эффект на сильно изогнутых поверхностях
- [ ] 2.1.6 LOD-поддержка: параметр `detail_level: f64` (0.0 = грубо, 1.0 = точно) → масштабирует `max_deviation`
- [ ] 2.1.7 Unit-тесты: сравнить площадь триангулированной сферы с аналитической (4πR²) при разных `max_deviation`
- [ ] 2.1.8 Тест: визуальная проверка — сфера при `max_deviation=0.001` должна быть гладкой, при `0.1` — гранёной

**Критерий приёмки:** `max_deviation` и `max_edge_length` реально контролируют плотность сетки. Площадь триангуляции сферы отличается от аналитической не более чем на `max_deviation * surface_area_ratio`.
